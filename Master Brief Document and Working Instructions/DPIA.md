## 1. Why this DPIA exists

The app processes personal data of children (target age 9–11) for educational purposes. Under UK GDPR Article 35, a DPIA is required when processing is likely to result in a high risk to individuals' rights, and specifically when processing children's data at scale. This document sets out what data is collected, why, how it's protected, and what the residual risks are.

## 2. What the app does

A React web app at https://11plus-prep.pages.dev/ that helps children practise for the GL Assessment 11+ entrance exam. It presents 6,682 multiple-choice questions across Maths, English, and Verbal Reasoning, and **tracks progress** so the child and parent can see strengths, weaknesses, and improvement over time. Each parent account maps to exactly one child profile.

## 3. Personal data processed

| Category | Data | Source | Storage |
|---|---|---|---|
| **Parent identity** | Email address, name (optional) | Clerk sign-up | Clerk (auth provider) |
| **Child identity** | Display name only (no surname) | Onboarding form | D1 database (Cloudflare) |
| **Educational data** | Quiz results, per-question answers, topic performance, streaks, mastery scores, achievement progress, lessons viewed | Generated by app use | D1 database |
| **Behavioural telemetry** | Error reports (error message, stack trace, URL, user agent, user's display name, timestamp) | ErrorBoundary → Worker | Workers KV (ring buffer of last 50 entries) |
| **Testing/QA data** | Question flags with category + note + submitter name | Testing Mode (dev only) | Workers KV |
| **Client cache** | Mirror of educational data for offline use | Auto-written on each save | Browser localStorage |

**No special category data** (health, biometric, political, religious, sexual orientation, etc.) is collected.
**No payment data** — the app is free.
**No location data** beyond the URL a user was on when an error fired.

## 4. Lawful basis

- **Parent's personal data (email, name):** Consent (UK GDPR Article 6(1)(a))
- **Child's personal data:** Parental consent for a child under 13, given during account setup (Article 6(1)(a) + Article 8)
- **Educational processing:** Also justified as performance of the service contract the parent enters into when creating the account (Article 6(1)(b))

Consent is obtained via an explicit checkbox during onboarding. The consent flow records date and opt-in choices.

## 5. Processing purposes

1. **Deliver the service:** serve questions, record answers, track progress, power the recommended-next-topic logic
2. **Personalise:** choose which questions a child sees next (spaced repetition, mastery-weighted selection)
3. **Give parents visibility:** Parent Dashboard surfaces strengths and weaknesses so parents can support their child
4. **Debug and improve:** error reports help fix crashes; testing flags help fix content bugs before they reach other users

No processing for advertising, profiling beyond educational use, or sale of data to third parties.

## 6. Retention

- **Account-level data:** retained for the duration of the account. Parents can export or delete the account at any time via the Account menu (the delete action is server-authoritative — data is removed from D1, localStorage is only cleared once the server confirms).
- **Error reports:** ring buffer of last 50 errors — older entries automatically drop out
- **Testing flags:** retained until manually resolved by Ben/Jacqui
- **Authentication data at Clerk:** governed by Clerk's retention policy (see Clerk's own DPA)

## 7. Data sharing / third parties

The app uses the following processors:

| Processor | What they do | Where data sits | DPA? |
|---|---|---|---|
| **Cloudflare** | Hosting (Pages), serverless compute (Workers), database (D1), KV store | Global — primarily EU/UK edge | Covered by Cloudflare's standard DPA |
| **Clerk** | Authentication + account management | US-based, with GDPR terms | Covered by Clerk's DPA |
| **Anthropic** | AI Tutor chat (Claude Haiku 4.5) — only called when a child uses the tutor feature | US-based, with zero-retention API | Covered by Anthropic's API terms |

**AI Tutor specifics:** When a child asks the AI Tutor a question, the question context (current quiz question + user message) is sent to Anthropic. No identifiers (name, email, account ID) are sent. Anthropic's API is configured for zero retention of prompts.

## 8. Security measures

- **Transport:** HTTPS everywhere (TLS 1.3 at Cloudflare's edge)
- **Auth:** Clerk-managed JWTs verified at the Worker; JWKS rotated via Clerk
- **Isolation:** D1 rows carry a `user_id` foreign key, queried only via the authenticated user's token. No cross-user reads possible at the API layer
- **At rest:** D1 and KV are encrypted by Cloudflare
- **Client side:** sensitive tokens not persisted to localStorage. Only cached educational data.
- **Secrets management:** Clerk publishable keys in frontend env (safe by design), Clerk secret + Anthropic API key in Worker secrets (never in source control)
- **Dependency hygiene:** `npm audit` run periodically; major framework upgrades tested before deploy
- **Error reporting rate limit:** client-side throttle (10 reports per session, deduped) prevents a misbehaving tab from DoSing the reporting endpoint or leaking repeated stack traces

## 9. Children's Code considerations (ICO Age-Appropriate Design Code)

| Principle | How we comply |
|---|---|
| **Best interests of the child** | Only data needed to deliver education; no behavioural advertising; progress data stays private to parent + child |
| **Data minimisation** | We don't collect surnames, dates of birth, location, school, or photos |
| **Transparency** | This DPIA + parent-facing one-pager explain what's collected in plain language |
| **Default settings** | Privacy-maximising by default; no social sharing, no public profiles, no third-party trackers |
| **Nudge techniques** | Gamification (streaks, prep points) is scoped to educational reinforcement; designed per the 14-doc research library with ethical guardrails (no dark patterns, no pay-to-win) |
| **Parental controls** | Parent Dashboard + account-level delete/export + no in-app purchases |
| **No profiling against the child's best interests** | Profiling is only used for spaced-repetition question selection, which is in the child's learning interest |

## 10. Risks and mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Unauthorised access to a child's progress data | Low | Low-medium | Clerk auth; per-account isolation in D1; JWT verification at every API call |
| Cloudflare or Clerk breach | Very low | High | Rely on processor's SOC 2 / ISO 27001 compliance; monitor their security advisories |
| Child's self-image harmed by seeing poor results | Low | Low | Results framed constructively; micro-lessons before quizzes; mistakes presented as "still to learn" not "failed" |
| Parental consent collected without genuine parental review | Low | Medium | Onboarding is done by the parent on their device; explicit consent checkbox; parents can delete data anytime |
| AI Tutor reveals answers or generates inappropriate content | Low | Medium | System prompts hardened; tutor refuses to give answers on interact screens; child-safe system prompt |
| Regression in an update corrupts progress data | Low | High | Smoke test gates every deploy; 358-test Jest suite; D1 writes atomic via batch; localStorage fallback |
| Too few users today for anomaly detection | High | Low | Soft launch with small group; error dashboard checked daily |

## 11. Individual rights

| Right | How exercised |
|---|---|
| **Access** | Export JSON from Account menu → "Export Data" |
| **Rectification** | Re-take a quiz (latest result supersedes earlier ones) or email ben@venortech.com for anything else |
| **Erasure** | "Delete Account" in Account menu — server-authoritative |
| **Portability** | Export JSON (machine-readable) |
| **Objection / withdraw consent** | Delete Account |
